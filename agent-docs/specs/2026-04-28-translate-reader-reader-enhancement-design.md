# translate-reader 阅读器增强与人工微调设计

日期：2026-04-28

## 1. 目标

在现有 `translate-reader` 首版闭环基础上，继续把产品收敛为一个“本地 Web 机翻阅读器”，而不是重型翻译工作台。

本阶段优先级固定为：

1. `A` 阅读器体验优先
2. `B` 章节上下文一致性
3. `C` 单页性能优化

目标效果：

- 默认一键生成可阅读的译图
- 原图 / 译图可快速切换对照
- 单个气泡翻得不好、排版超框、阅读不顺时，可直接手动调整
- 调整后能立即重渲染，不打断阅读流程
- 手动修正结果可反向影响后续页翻译，减少称呼、术语、语气漂移

## 2. 产品定位

本项目的核心形态是“阅读器 + 轻编辑器”。

用户主流程应当是：

1. 导入一个章节目录
2. 选择页并触发默认整页翻译
3. 在阅读视图中对照原图和译图
4. 若某一框翻译或排版不理想，直接在当前页内调整
5. 保存当前页修正结果
6. 后续页翻译默认继承前文上下文和已确认术语

这意味着系统优化目标不是“最大化一次性全自动成功率”，而是“尽快给出一个能读的初稿，并把人工修正成本压低”。

## 3. 当前基线

截至 2026-04-28，系统已具备：

- 本地目录导入
- 单页 `检测 -> OCR -> 翻译 -> 擦字 -> 写字`
- 原图 / 译图切换
- 检测误框首层过滤
- 单页重做擦字 / 重做写字
- Saber 自动字号桥接

当前单页真实耗时基线：

- 常见区间约 `28s - 60s / 页`
- 2026-04-28 在 `翻译测试日漫/单话测试/011.jpg` 上，从开始运行到译图生成实测约 `32.06s`

这组时间应被视为后续优化的基准线，而不是暂时现象。

## 4. 本阶段范围

### 4.1 范围内

- 阅读器页列表与当前页工作区
- 原图 / 译图对照切换
- 单气泡人工微调
- 单气泡与整页重渲染
- 页面级手动修改持久化
- 章节级上下文一致性输入
- 单页阶段耗时记录与展示

### 4.2 暂不纳入

- 完整 Saber 设置中心
- 自由多边形笔刷编辑器
- 全章节后台任务编排系统
- 复杂术语库独立管理后台
- 书架、多会话、多项目管理

## 5. 两种设计路线

### 方案 A：阅读器优先增强（推荐）

在现有 `translate-reader` 基础上，补齐阅读器工作区、单框编辑、上下文一致性和性能度量。

优点：

- 复用现有目录导入、缓存和整页流水线
- 范围可控，产品边界清晰
- 迭代速度快，最符合“先 A 再 B 再 C”

缺点：

- 需要自己补轻量编辑器层
- 不能直接获得 Saber 的完整工作台能力

### 方案 B：继续大幅迁移 Saber 编辑工作台

把 Saber 现有编辑链路、样式控制和更多交互直接迁入。

优点：

- 可较快获得更完整的编辑能力
- 与 Saber 功能更接近

缺点：

- 容易演变为另一个 Saber 分支
- 阅读器产品定位会被稀释
- 调试与维护复杂度显著上升

结论：

本阶段继续采用 `方案 A`，但选择性迁移 Saber 的高价值能力，而不迁其完整工作台。

## 6. 最值得继续迁移的 Saber 能力

优先级从高到低如下：

1. `re_render_single_bubble`
   - 支撑“只修这一框”
   - 比整页重渲染更符合阅读器微调场景

2. `bubble_states`
   - 统一保存每框的字号、方向、对齐、描边、偏移、颜色等状态
   - 是人工微调与结果持久化的核心

3. `calculate_auto_font_size`
   - 已接通
   - 后续应与手动字号/偏移共同工作，而不是互斥

4. `parallel/color`
   - 已接桥接，但当前环境缺 `ocr_48px` 模型，实际会降级为空颜色
   - 后续补模型后可启用自动字色 / 底色

5. `lineSpacing / textAlign / stroke / rotation / position`
   - 这些参数直接决定“顺眼舒服”的最终观感

## 7. 阅读器增强设计

### 7.1 页面结构

界面采用固定三栏结构：

- 左侧：页列表 / 状态
- 中间：阅读画布
- 右侧：当前页与当前气泡编辑面板

中间画布需要同时支持：

- 显示原图
- 显示译图
- 框层叠加
- 当前选中框高亮

### 7.2 核心交互

第一阶段必须支持：

- 原图 / 译图切换
- 选择当前气泡
- 编辑译文文本
- 拖拽 / 缩放气泡框
- 调整字号、方向、行距、对齐、描边、偏移
- 单框重排版
- 当前页重做擦字
- 当前页重做写字

系统的默认策略应是：

- 先自动给出尽量可用的结果
- 再允许用户最小成本修正

### 7.3 结果持久化

单页缓存不再只保存自动结果，还应保存：

- `bubbleStates`
- `manualOverrides`
- `geometryEdits`
- `renderTiming`
- `contextSnapshot`

这样重新打开页面时，用户能回到上次编辑状态，而不是从自动结果重新开始。

## 8. 数据与接口设计

### 8.1 页面结果结构增强

在现有 `result.json` 中补充：

- `bubbleStates`
  - 每框最终样式状态
- `manualEdited`
  - 当前页是否被人工修改
- `timings`
  - `detect / ocr / translate / inpaint / render / total`
- `contextInputs`
  - 当前页翻译使用的上下文摘要与术语快照

### 8.2 新增接口方向

建议新增以下接口：

- `POST /api/pipeline/rerender-bubble`
  - 只重排当前气泡

- `POST /api/pipeline/update-bubble`
  - 更新某框文本、几何和样式状态

- `POST /api/pipeline/save-page-edits`
  - 保存当前页全部人工调整

- `GET /api/page/<id>/timings`
  - 返回当前页阶段耗时

- `GET /api/chapter/context`
  - 返回当前章节术语与上下文摘要

### 8.3 接口原则

- 单框编辑与整页编辑共用同一套 `bubbleStates`
- 所有人工调整都先写缓存，再决定是否立即重渲染
- 单框重渲染默认不重新翻译、不重新擦字

## 9. 上下文一致性设计

### 9.1 问题定义

当前系统翻译单位仍以单页为主，容易出现：

- 人名称呼前后不一致
- 术语翻译漂移
- 角色语气在相邻页跳变

### 9.2 目标策略

章节级一致性不做成复杂知识库，先做“轻上下文”：

- 使用前若干页已确认译文摘要
- 维护一份章节内术语 / 人名 / 称呼表
- 手动修正过的译文优先级高于自动译文

翻译输入应包含：

- 当前页原文列表
- 前文关键上下文摘要
- 已锁定术语映射
- 已确认角色称呼

### 9.3 一致性优先级

当自动翻译与人工修正规则冲突时：

1. 当前页手动修正
2. 章节术语表
3. 邻近页上下文
4. 默认翻译模型输出

## 10. 性能设计

### 10.1 当前目标

现阶段不先硬做复杂优化，但必须先把性能可见化。

每页都应记录：

- 检测耗时
- OCR 耗时
- 翻译耗时
- 擦字耗时
- 写字耗时
- 总耗时

### 10.2 后续优化方向

在 `C` 阶段再做：

- 单框编辑只重做单框写字
- 文本修改不重跑检测 / OCR
- 页级缓存命中时跳过已完成步骤
- 章节阅读预热下一页检测 / OCR

### 10.3 现实判断

“单页从开始运行到译图出来的时间”是阅读器体验的一部分。

但在当前优先级下：

- 只要能稳定在几十秒内得到可编辑初稿
- 并且用户后续微调不需要重跑整页完整流程

阅读体验就能先成立。

## 11. 分阶段里程碑

### A1：阅读器可用

- 阅读器三栏布局稳定
- 原图 / 译图切换
- 当前页状态与耗时展示
- 选择当前框并查看其原文 / 译文

### A2：人工微调闭环

- 单框文本编辑
- 单框框体拖拽 / 缩放
- 单框样式编辑
- 单框重排版
- 页面编辑结果持久化

### B1：章节上下文一致性

- 章节上下文摘要
- 术语 / 称呼锁定
- 翻译 prompt 注入前文上下文
- 手动修正反哺后续页

### C1：性能优化

- 阶段耗时看板
- 单框重渲染替代整页重跑
- 缓存命中策略细化
- 预热与跳步优化

## 12. 风险

### 12.1 编辑器复杂度膨胀

如果一开始追求任意自由编辑，会迅速变成完整桌面排版器。

控制方式：

- 第一阶段只做框级编辑
- 不做自由画笔和复杂多边形编辑

### 12.2 上下文输入过重

如果把大量前文原文直接塞给模型，成本和延迟都会上升。

控制方式：

- 只传摘要、术语和邻近页关键内容
- 不做全章原文拼接

### 12.3 颜色能力误判

当前颜色桥接虽已接入，但本机没有 48px 颜色模型文件。

控制方式：

- 当前明确视为“可降级能力”
- 不把自动字色作为第一阶段验收条件

## 13. 验收标准

当本阶段完成时，系统应满足：

- 用户能导入整个章节目录并逐页阅读
- 单页能默认生成可阅读的译图
- 用户能在当前页直接调整某一框的文本、框体和样式
- 调整后能立即看到新的译图结果
- 原图 / 译图可快速对照
- 手动修正结果能够保存并在重新打开后复现
- 翻译结果可逐步具备章节级称呼和术语一致性

## 14. 结论

下一阶段不应继续把精力平均分配在所有功能上，而应严格按以下顺序推进：

1. 把 `translate-reader` 做成真正顺手的阅读器
2. 把人工修正闭环做短
3. 再让上下文一致性提升整体翻译质量
4. 最后再围绕真实耗时数据压性能

这条路径最符合当前产品目标，也最能避免项目重新膨胀成完整 Saber 复刻。
